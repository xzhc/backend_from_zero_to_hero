## 1. 什么是 Channel

Channel（通道）是 Go 语言中 goroutine 之间**通信**的管道。它遵循 Go 的核心哲学：

> **"不要通过共享内存来通信，而要通过通信来共享内存。"**

> 📖 **新手友好补充**
>
> ### 🔍 生活类比：快递传送带
>
> 把 channel 想象成**快递公司的传送带**：
> - **发送方（goroutine A）** 把包裹（数据）放到传送带上
> - **接收方（goroutine B）** 从传送带上取走包裹
> - 传送带保证了**顺序**和**安全性**，发送方和接收方不需要直接接触
>
> 如果传送带满了（有缓冲通道塞满），发送方就要等待；  
> 如果传送带上没有包裹，接收方就要等待。
>
> ### 💡 为什么需要 Channel？
>
> 多个 goroutine 之间需要传递数据时，最危险的做法是直接共享变量（见 Mutex 那章）。Channel 提供了一种更安全、更优雅的方式：**通过传递数据本身来同步状态**，避免了竞态条件，代码也更清晰。

## 2. 基本用法

### 创建 Channel

```go
// 创建一个传递 int 类型的无缓冲通道
ch := make(chan int)

// 创建一个传递 string 类型、缓冲大小为 3 的有缓冲通道
ch2 := make(chan string, 3)
```

### 发送与接收

```go
ch <- 42      // 向通道发送数据（箭头方向 → 通道）
value := <-ch // 从通道接收数据（箭头方向 ← 通道，赋值给 value）
<-ch          // 接收但丢弃数据（只关心"有没有"，不关心值）
```

> 📖 **新手友好补充**
>
> ### 📝 最小可运行示例：goroutine 之间传数据
>
> ```go
> package practice
>
> import "testing"
>
> func TestChannelBasic(t *testing.T) {
>     ch := make(chan int) // 创建无缓冲通道
>
>     // 启动一个 goroutine 发送数据
>     go func() {
>         ch <- 100 // 发送 100 到通道（这里会阻塞，直到有人接收）
>     }()
>
>     value := <-ch // 主 goroutine 接收数据（这里会阻塞，直到有人发送）
>     t.Log("收到:", value) // 输出: 收到: 100
> }
> ```
>
> **关键点**：`ch <- 100` 和 `<-ch` 是一对，缺了任何一个，程序就会永远等待（死锁）。

## 3. 无缓冲 vs 有缓冲 Channel

### 无缓冲 Channel（`make(chan T)`）

- **发送**会阻塞，直到有接收方准备好
- **接收**会阻塞，直到有发送方发送数据
- 发送和接收**同步**发生（必须同时准备好）

### 有缓冲 Channel（`make(chan T, N)`）

- 内部有一个大小为 N 的队列
- **发送**只在队列满时才阻塞
- **接收**只在队列空时才阻塞
- 发送和接收可以**异步**进行

> 📖 **新手友好补充**
>
> ### 🔍 生活类比：咖啡店取餐台
>
> - **无缓冲**：顾客（发送方）点完咖啡后，必须站在原地等到咖啡师（接收方）直接亲手递给他才能离开。双方必须"面对面同步"。
> - **有缓冲（容量=3）**：咖啡师做好咖啡可以放到取餐台（最多放3杯），顾客随时来取，不用面对面。但取餐台满了，咖啡师就得等；取餐台空了，顾客就得等。
>
> ### 📝 对比示例
>
> ```go
> func TestBufferedVsUnbuffered(t *testing.T) {
>     // ✅ 有缓冲通道：发送方不会立即阻塞（只要未满）
>     buffered := make(chan int, 3)
>     buffered <- 1 // 不阻塞，放入缓冲区
>     buffered <- 2 // 不阻塞
>     buffered <- 3 // 不阻塞（缓冲区还剩1个位置）
>     // buffered <- 4 // ❌ 这里会死锁！缓冲区满了
>
>     t.Log(<-buffered) // 1
>     t.Log(<-buffered) // 2
>     t.Log(<-buffered) // 3
>
>     // ❌ 无缓冲通道：在同一个 goroutine 里发送后立刻接收 → 死锁
>     // unbuffered := make(chan int)
>     // unbuffered <- 1  ← 这里会永远阻塞，因为没有其他 goroutine 来接收
> }
> ```
>
> ### ⚠️ 新手常见误区
>
> **误区：以为有缓冲就不会阻塞**  
> 缓冲只是给你一些"余量"，一旦缓冲区满了，发送方照样阻塞。缓冲大小 ≠ 无限容量。

## 4. Channel 的关闭与遍历

### 关闭 Channel

```go
close(ch) // 关闭通道，告诉接收方：不会再有新数据了
```

关闭后：
- 还可以**接收**已经在通道里的数据
- 不能再**发送**数据（会 panic）
- 接收完已有数据后，继续接收会得到零值

### 检测通道是否已关闭

```go
value, ok := <-ch
if !ok {
    // ok 为 false，说明通道已关闭且没有数据了
}
```

### 用 for range 遍历

```go
for v := range ch {
    // 循环接收通道中的数据，直到通道被关闭
    fmt.Println(v)
}
```

> 📖 **新手友好补充**
>
> ### 📝 最小可运行示例：生产者-消费者模式
>
> ```go
> func TestChannelClose(t *testing.T) {
>     ch := make(chan int, 5)
>
>     // 生产者 goroutine：发送数据后关闭通道
>     go func() {
>         for i := 1; i <= 5; i++ {
>             ch <- i
>         }
>         close(ch) // 发送完毕，关闭通道，通知消费者没有新数据了
>     }()
>
>     // 消费者：用 for range 接收，通道关闭后自动退出循环
>     for v := range ch {
>         t.Log("收到:", v)
>     }
>     // 输出：收到: 1  收到: 2  收到: 3  收到: 4  收到: 5
> }
> ```
>
> ### ⚠️ 新手常见误区
>
> **误区 1：重复关闭通道 → panic**
> ```go
> close(ch)
> close(ch) // ❌ panic: close of closed channel
> ```
>
> **误区 2：向已关闭的通道发送数据 → panic**
> ```go
> close(ch)
> ch <- 1 // ❌ panic: send on closed channel
> ```
>
> **误区 3：接收方关闭通道**  
> 关闭通道的责任应该由**发送方**承担，因为只有发送方知道"数据发送完了"。接收方关闭通道容易造成发送方 panic。
>
> **误区 4：忘记关闭通道，`for range` 永远不退出**
> ```go
> // ❌ 危险：如果 ch 从未关闭，这个循环会永远阻塞
> for v := range ch {
>     fmt.Println(v)
> }
> ```

## 5. select 多路复用

`select` 可以同时监听多个 channel，哪个先就绪就执行哪个分支。

```go
select {
case v := <-ch1:
    // ch1 有数据
case v := <-ch2:
    // ch2 有数据
case ch3 <- value:
    // 成功向 ch3 发送数据
default:
    // 所有 channel 都没有就绪时执行（非阻塞模式）
}
```

> 📖 **新手友好补充**
>
> ### 🔍 生活类比：同时等多个外卖
>
> 你同时点了3家外卖（3个 goroutine），谁先送到你就先吃谁的。`select` 就是这个"哪个先到就先处理哪个"的机制。
>
> ### 📝 最小可运行示例：超时控制
>
> ```go
> func TestSelectTimeout(t *testing.T) {
>     ch := make(chan string)
>
>     // 模拟一个慢操作（2秒后才返回结果）
>     go func() {
>         time.Sleep(2 * time.Second)
>         ch <- "终于完成了"
>     }()
>
>     select {
>     case result := <-ch:
>         t.Log("收到结果:", result)
>     case <-time.After(1 * time.Second): // 等 1 秒，超时就放弃
>         t.Log("超时！放弃等待")
>     }
>     // 输出: 超时！放弃等待
> }
> ```
>
> ### ⚠️ 新手常见误区
>
> **误区：select 没有 default，所有 case 都没就绪 → 永远阻塞**
> ```go
> // ❌ 如果 ch1 和 ch2 都没有数据，这里会永远等待
> select {
> case v := <-ch1:
>     fmt.Println(v)
> case v := <-ch2:
>     fmt.Println(v)
> }
>
> // ✅ 加 default 实现非阻塞检查
> select {
> case v := <-ch1:
>     fmt.Println(v)
> default:
>     fmt.Println("暂时没有数据，先去做别的事")
> }
> ```
>
> **误区：多个 case 同时就绪时，select 随机选一个（不是按顺序）**
> ```go
> // 如果 ch1 和 ch2 都有数据，Go 会随机选择其中一个执行，不保证顺序
> select {
> case <-ch1: ...
> case <-ch2: ...
> }
> ```

## 6. Channel 方向（只读 / 只写）

函数参数中可以限定 channel 的方向，增加代码安全性：

```go
func producer(out chan<- int) { // out 只能发送（只写）
    out <- 42
}

func consumer(in <-chan int) { // in 只能接收（只读）
    v := <-in
    fmt.Println(v)
}
```

> 📖 **新手友好补充**
>
> ### 💡 为什么要限定方向？
>
> 如果 producer 函数拿到了双向通道，它可能不小心从通道读数据，或者关闭通道（这是接收方不该做的事）。限定方向让编译器帮你检查，避免误操作。
>
> ```go
> func producer(out chan<- int) {
>     <-out // ❌ 编译错误：out 是只写通道，不能接收
>     close(out) // ✅ 发送方可以关闭通道
> }
>
> func consumer(in <-chan int) {
>     in <- 1   // ❌ 编译错误：in 是只读通道，不能发送
>     close(in) // ❌ 编译错误：只读通道不能关闭
> }
> ```

## 7. 常见 Channel 模式

### 模式一：Done 信号（通知退出）

```go
done := make(chan struct{}) // 空结构体，不占内存，只用于信号

go func() {
    // ... 做一些工作 ...
    close(done) // 发出"完成"信号
}()

<-done // 主 goroutine 等待完成信号
```

### 模式二：Fan-Out（一个生产者，多个消费者）

```go
ch := make(chan int, 10)

// 一个生产者
go func() {
    for i := 0; i < 10; i++ {
        ch <- i
    }
    close(ch)
}()

// 多个消费者，都从同一个 ch 读取
for i := 0; i < 3; i++ {
    go func(id int) {
        for v := range ch {
            fmt.Printf("消费者 %d 处理: %d\n", id, v)
        }
    }(i)
}
```

### 模式三：Pipeline（流水线）

```go
// 阶段一：生成数据
func generate(nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        for _, n := range nums {
            out <- n
        }
        close(out)
    }()
    return out
}

// 阶段二：处理数据（每个数乘以2）
func double(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        for v := range in {
            out <- v * 2
        }
        close(out)
    }()
    return out
}

// 使用：generate → double → 输出
func TestPipeline(t *testing.T) {
    ch := generate(1, 2, 3, 4, 5)
    doubled := double(ch)
    for v := range doubled {
        t.Log(v) // 2, 4, 6, 8, 10
    }
}
```

## 8. Channel 关键要点总结

| 特性 | 无缓冲 Channel | 有缓冲 Channel |
|------|---------------|---------------|
| 创建 | `make(chan T)` | `make(chan T, N)` |
| 发送阻塞 | 没有接收方时立即阻塞 | 缓冲区满时才阻塞 |
| 接收阻塞 | 没有发送方时立即阻塞 | 缓冲区空时才阻塞 |
| 同步性 | 强同步（发送和接收必须同时就绪） | 异步（有缓冲余量时可以不同时） |
| 适用场景 | goroutine 之间的同步点 | 异步任务队列、限流 |

**核心原则：**
1. ✅ 由**发送方**负责 `close(ch)`，不要由接收方关闭
2. ✅ 使用 `for range` 遍历通道，关闭时自动退出
3. ✅ 使用 `select` + `ctx.Done()` 实现优雅退出
4. ✅ 限定函数参数的通道方向（`chan<-` / `<-chan`），增加安全性
5. ⚠️ 不要关闭未初始化的 nil channel（会 panic）
6. ⚠️ 不要重复 close，不要向已关闭的 channel 发送数据
